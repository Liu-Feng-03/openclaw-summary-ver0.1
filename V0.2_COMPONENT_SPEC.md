## 1. Purpose

This document defines the main product components for v0.2 at the behavior and structure level.

It answers:
- what components exist
- what each component is for
- what fields it contains
- how it behaves
- where it should be used

This spec is intended to reduce implementation ambiguity and improve visual consistency.

---

## 2. Component Principles

### P1. Components must express product meaning, not just layout
A component should reveal whether it is for routing, preview, reading, diagnosis, or progression.

### P2. Components must preserve hierarchy
Title, summary, metadata, and CTA should not visually fight each other.

### P3. Components must support consistency
The same component family should behave predictably across modules.

### P4. Components must still allow module flavor
Shared structure should not eliminate module identity.

---

## 3. Core Component Inventory

The core v0.2 component families are:

1. TopNav
2. ExplorerNav
3. Breadcrumb
4. PageHeader
5. MetadataRow
6. RouteCard
7. ExpandableTopicCard
8. AccordionGroup / AccordionItem
9. QuickPathBlock
10. LearningPathStep
11. NextStepBlock
12. RelatedPagesBlock
13. PrevNextNav
14. SummaryBox / QuickAnswerBox
15. ModuleBridgeBlock

---

## 4. `TopNav`

### Role
Global product navigation.

### Used on
- homepage
- module pages
- detail pages

### Required content
- brand / home link
- top-level modules
- active module state

### Behavior
- stays simple
- no heavy deep hierarchy inside top nav
- mobile collapses cleanly

### Notes
Detailed hierarchy belongs in ExplorerNav, not TopNav.

---

## 5. `ExplorerNav`

### Role
Persistent spatial navigation.

### Used on
- module pages (desktop)
- detail pages (desktop)
- drawer pattern on mobile

### Required content
- home entry
- top-level modules
- visible current module
- visible sibling pages / child pages
- current page highlight

### Behavior
- module groups expandable/collapsible
- current module expanded by default
- current page unmistakably active

### Why it matters
This is the main component that solves the “I still don’t know where I am” problem.

---

## 6. `Breadcrumb`

### Role
Local hierarchy confirmation.

### Structure
- item list
- optional separators

### Typical pattern
首页 / 模块 / 页面

### Used on
- module pages
- detail pages

### Behavior
- clickable up to the current page
- current page is non-clickable text

---

## 7. `PageHeader`

### Role
Page-level orientation and context.

### Variants
- HomepageHeader
- ModuleHeader
- DetailHeader

### Shared fields
- label / eyebrow
- title
- subtitle / summary
- optional tags / metadata
- optional CTA area

### Rules
- each variant must feel different in hierarchy
- homepage is strongest
- module is lighter
- detail is calmest

---

## 8. `MetadataRow`

### Role
Compact decision-support metadata.

### Typical fields
- difficulty
- reading time
- audience
- tags

### Used on
- module cards
- detail headers
- related/route cards where useful

### Rules
- metadata must not dominate the component
- use only fields that genuinely help decisions
- avoid cluttering every component with full metadata

---

## 9. `RouteCard`

### Role
Entry component for route selection.

### Used on
- homepage entry modes
- quick route sections
- fast path blocks

### Required fields
- title
- one-sentence description
- href or target action

### Optional fields
- audience
- estimated effort
- route type

### Behavior
- direct navigation
- can have hover preview emphasis

### Visual rule
Should feel more like “choose a path” than “open a content item.”

---

## 10. `ExpandableTopicCard`

### Role
Primary module-page preview component.

### Used on
- module pages
- possibly grouped content sections

### Collapsed state must include
- title
- summary
- metadata row

### Expanded state should include
- why it matters
- who should read it
- prerequisite pages
- related pages
- common misunderstanding or use case
- CTA to full page

### Behavior
- click/expand without immediate hard navigation
- open full page only on explicit CTA or title action if chosen

### Important rule
This component exists specifically to reduce jump anxiety.

---

## 11. `AccordionGroup` / `AccordionItem`

### Role
Category-based reveal component.

### Primary use
- FAQ module
- secondary grouped navigation inside some modules

### AccordionItem should support
- title
- short preview
- expanded explanation
- optional likely cause / confidence note / route suggestion

### Behavior
- single-open or multi-open mode depending on section needs
- FAQ likely benefits from single-open per category in many contexts

---

## 12. `QuickPathBlock`

### Role
Fast route shortcuts for impatient or narrowly-focused users.

### Used on
- homepage
- module pages
- possibly some detail pages

### Structure
- title
- 2–4 short route options
- each option uses RouteCard or compact variant

### Example titles
- 如果你只有 10 分钟
- 如果你只想尽快跑起来
- 如果你已经卡住了

---

## 13. `LearningPathStep`

### Role
Represent one stage in the guided learning path.

### Used on
- homepage learning path
- getting started
- setup progression where adapted

### Fields
- step label / number
- title
- outcome
- time / effort
- recommended pages

### Behavior
- expandable preferred
- should reveal more detail before navigation

---

## 14. `NextStepBlock`

### Role
Strongly guide the next recommended move.

### Used on
- detail pages
- module pages where helpful

### Required content
- one primary recommendation
- optional secondary routes
- short explanation for each

### Behavior
- one option should visually read as the recommended continuation

### Important rule
This is for guided progression, not for general browsing.

---

## 15. `RelatedPagesBlock`

### Role
Support lateral learning expansion.

### Used on
- detail pages
- potentially module pages

### Required content
- related page title
- one-sentence reason for relation

### Behavior
- should not visually compete with NextStepBlock

### Difference from NextStepBlock
- Related = lateral exploration
- Next Step = preferred continuation

---

## 16. `PrevNextNav`

### Role
Sequential continuity.

### Used on
- detail pages where order matters

### Required content
- previous page
- next page

### Behavior
- should follow module-defined sequence
- should not be purely chronological or alphabetical unless intentionally designed that way

---

## 17. `SummaryBox` / `QuickAnswerBox`

### Role
Fast comprehension layer.

### Used on
- detail pages
- possibly some overview pages

### Required content
- short answer
- why it matters
- who this is for
- optional prerequisite / next step

### Why it matters
Especially useful for:
- shared-entry visitors
- impatient users
- mobile readers

---

## 18. `ModuleBridgeBlock`

### Role
Prevent module silo behavior.

### Used on
- module pages
- some detail pages

### Required content
- one or more cross-module suggestions
- explanation of why the user may want to go there next

### Example
- Concepts → Setup
- Setup → FAQ
- FAQ → Concepts

---

## 19. Explorer Item Spec

Because navigation is now a central problem, explorer items need explicit rules.

### Explorer item fields
- label
- href
- level
- active state
- expandable children (optional)

### States
- default
- hover
- expanded
- active module
- active page

### Rule
The active page state must be stronger than hover and stronger than expanded group state.

---

## 20. Card Spacing and Alignment Rules

### Shared rules for card-like components
- stable padding
- stable title-to-summary gap
- stable summary-to-metadata gap
- CTA zone separated from content area

### Equal-height groups
Apply to:
- route cards
- module exploration cards
- shareable links

### Flexible-height groups
Apply to:
- expandable topic cards
- accordion items
- some related content cards

---

## 21. Content Responsibility by Component

### RouteCard
Helps user choose direction.

### ExpandableTopicCard
Helps user preview a topic.

### AccordionItem
Helps user inspect categorized questions or grouped content.

### SummaryBox
Helps user understand fast.

### NextStepBlock
Helps user continue.

### RelatedPagesBlock
Helps user expand laterally.

This distinction must stay clear.

---

## 22. Component Anti-Patterns

Avoid these:

### Anti-pattern 1
Using the same visual card for route selection, topic preview, and FAQ diagnosis without meaningful differentiation.

### Anti-pattern 2
Letting title, metadata, and CTA all compete equally in one component.

### Anti-pattern 3
Turning every component into a giant clickable block that forces navigation immediately.

### Anti-pattern 4
Using decorative badges that add no decision value.

### Anti-pattern 5
Skipping component state design and relying on browser defaults or accidental styling.

## 23. Short Handoff Summary

If another process reads only this file, the key message is:

> v0.2 needs stronger component semantics, not just prettier boxes.

The most important components to get right are:
- ExplorerNav
- PageHeader variants
- ExpandableTopicCard
- AccordionGroup
- SummaryBox
- NextStepBlock

These components together define whether the product feels like:
- a real learning system
or
- a set of attractive but disconnected pages.
