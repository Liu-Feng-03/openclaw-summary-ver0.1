# V0.2_NAVIGATION_SYSTEM.md — openclaw-summary

Last updated: 2026-03-31
Status: navigation and spatial-orientation specification for v0.2

---

## 1. Why this document exists

The current v0.2 direction already improves routing and page structure, but one critical gap remains:

> Users still do not strongly feel where they are inside the whole learning product.

The product currently has more **route sense** than **map sense**.

That is not enough.

A strong learning product needs both:
- **route sense**: what should I do next?
- **map sense**: where am I in the total system?

This document defines the navigation system that should solve that gap.

---

## 2. Navigation goals

The navigation system should help users answer five questions at all times:

1. What product am I inside?
2. Which major module am I in?
3. Which page am I on?
4. What else exists around me?
5. Where can I go next without getting lost?

Success means:
- users no longer feel the site is a sequence of disconnected pages
- users can learn through guided flow or free exploration without losing orientation
- users can move laterally across the knowledge system when needed

---

## 3. Core navigation model

v0.2 should use a **dual-navigation model**:

### Layer A — Guided navigation
This supports structured learning.
Examples:
- next step
- previous / next page
- recommended route
- quick-start path

### Layer B — Spatial navigation
This supports map awareness.
Examples:
- explorer / left-side navigation tree
- breadcrumb
- active module state
- current-page highlight

The key design rule is:

> Guided navigation tells users how to move forward.
> Spatial navigation tells users where they currently are.

Both must coexist.

---

## 4. Navigation surfaces in v0.2

The product should have four navigation surfaces.

### N1. Top navigation
Purpose:
- product-level orientation
- fast switching across top-level modules

### N2. Left explorer / learning navigator
Purpose:
- show the whole nearby structure
- show current location
- support module-level and page-level spatial awareness

### N3. Breadcrumb
Purpose:
- provide quick upward orientation
- show exact page hierarchy

### N4. In-content progression blocks
Purpose:
- next step
- previous / next
- related pages
- route shortcuts

---

## 5. Left Explorer / Learning Navigator

This is the most important new navigation addition.

### 5.1 Role
The left explorer should be the product’s main spatial navigation surface.

It should not replace guided reading.
It should support it.

### 5.2 What it should help users feel
- I know what module I am in
- I can see nearby pages
- I can see what else exists in this section
- I can jump directly when I want to
- I am not trapped inside the current page

### 5.3 Recommended behavior
The explorer should be:
- visible on module pages and detail pages on desktop
- collapsible on smaller screens
- aware of current module and current page
- able to expand/collapse module sections

### 5.4 Information hierarchy in the explorer
Recommended structure:

1. Product home
2. Top-level modules
3. Module subpages / key child pages
4. Current page highlight

### 5.5 Explorer hierarchy example

- 首页
- 入门引导
  - 什么是 OpenClaw
  - 如何学习
  - 首轮路线图
- 核心概念
  - 整体架构
  - Session
  - Gateway
  - Skills
  - Tools
  - Memory
- 配置与部署
  - Requirements
  - Install
  - Configuration
  - Start Gateway
  - First Message
  - Troubleshooting
- 常见问题
  - Setup
  - Connectivity
  - Files & Memory
  - Skills & Tools
  - Security & Behavior
- 实际场景
- 进阶主题

### 5.6 Explorer display rules

#### On homepage
- do **not** show the full explorer in its heavy form
- homepage should stay cleaner and more route-driven
- homepage may show a lighter “knowledge map / quick module navigator” instead

#### On module pages
- show explorer expanded to the current module
- highlight the current module
- reveal major children inside that module

#### On detail pages
- show explorer expanded to the current module
- highlight the exact current page
- nearby sibling pages should be visible

### 5.7 Explorer interaction rules

- clicking a module title opens that module overview page
- clicking a chevron expands/collapses module children
- current page is highlighted
- current module remains expanded by default
- other modules may remain collapsed unless user opens them

### 5.8 What the explorer should not do
- should not become an overwhelming full sitemap dump
- should not show every tiny page if that harms scanability
- should not replace breadcrumb or next-step logic

## 6. Top Navigation

### 6.1 Role
The top nav is for product-level movement, not for detailed page hierarchy.

### 6.2 Must include
- brand / home link
- top-level modules
- active module highlight

### 6.3 Should not do too much
- do not overload with deep dropdown trees in first serious v0.2 design
- the explorer should handle the deeper structure

### 6.4 Mobile behavior
- top nav should collapse to a compact pattern
- explorer access should be available through a drawer or toggle

---

## 7. Breadcrumb System

### 7.1 Role
Breadcrumb provides local hierarchy confirmation.

### 7.2 Standard pattern
- 首页 / 模块 / 页面

Examples:
- 首页 / 核心概念 / 整体架构
- 首页 / 配置与部署 / Start Gateway
- 首页 / 常见问题 / Connectivity

### 7.3 Rules
- show breadcrumb on module pages and detail pages
- homepage does not need breadcrumb
- breadcrumb labels should be short, readable, and stable

### 7.4 Why breadcrumb alone is not enough
Breadcrumb tells users where they are in hierarchy.
It does not show the surrounding map.
That is why explorer must also exist.

---

## 8. Guided Progression Blocks

These should remain in the system even after the explorer is added.

### 8.1 Next Step Block
Purpose:
- tell users the recommended next move

### 8.2 Previous / Next Navigation
Purpose:
- allow sequence-based reading where sequence matters

### 8.3 Related Pages Block
Purpose:
- allow lateral exploration without losing relevance

### 8.4 Quick Path Blocks
Purpose:
- let users bypass long browsing when their intent is narrow

Important rule:

> Explorer is for map awareness.
> Progression blocks are for flow.

Do not merge them into one unclear thing.

---

## 9. Navigation by Page Type

### 9.1 Homepage
Navigation priorities:
1. top nav
2. route selection
3. light knowledge map / quick module navigator

Homepage should not show the full heavy left explorer.
Otherwise it will feel like a documentation shell too early.

### 9.2 Module Page
Navigation priorities:
1. top nav
2. breadcrumb
3. left explorer
4. in-page route blocks
5. expandable module content

Module pages are the place where explorer becomes strongly valuable.

### 9.3 Detail Page
Navigation priorities:
1. top nav
2. breadcrumb
3. left explorer
4. page header
5. next-step / related / prev-next

Detail pages need the strongest orientation support.

## 10. Explorer Content Strategy

The explorer should not mirror the entire raw sitemap equally.
It should present content in a more useful product hierarchy.

### 10.1 Priority within modules
Each module should emphasize:
- key entry pages
- core learning pages
- important branches

### 10.2 Optional secondary pages
Some pages can be tucked under “more” or visually secondary grouping if needed.

### 10.3 Module grouping suggestions

#### Getting Started
- 什么是 OpenClaw
- 如何学习
- 首轮路线图

#### Concepts
Core first:
- 整体架构
- Session
- Gateway
- Skills
- Tools
- Memory
Secondary later:
- Agent
- Node
- Dashboard
- Heartbeat / Cron
- Subagents
- ACP

#### Setup
- Requirements
- Install
- Configuration
- Start Gateway
- First Message
- Troubleshooting

#### FAQ
- Setup
- Connectivity
- WS / HTTP Fallback
- Files & Memory
- Skills & Tools
- Security & Behavior

This allows the explorer to stay useful without becoming visually noisy.

---

## 11. Visual and Interaction Rules for Explorer

### 11.1 Visual states needed
- default
- hover
- active module
- active page
- expanded group
- collapsed group

### 11.2 Active state behavior
- active page must be unmistakable
- active module should remain visually clear even if the user scrolls

### 11.3 Density rules
- explorer text must remain compact
- rows should be scannable
- indentation should show hierarchy clearly
- too many nested levels should be avoided in v0.2

### 11.4 Sticky behavior
Recommended:
- explorer stays sticky on desktop where viewport allows
- it should remain visible while content scrolls

This strongly improves “where am I?” confidence.

---

## 12. Mobile Navigation Pattern

### 12.1 Problem
A permanent left explorer does not fit well on small screens.

### 12.2 Recommended mobile solution
Use a **Navigation Drawer / Learning Map Drawer**.

### 12.3 Drawer should include
- current module
- current page
- module tree
- quick jump to sibling pages

### 12.4 Trigger label
Something clearer than generic “Menu” is preferable.
Examples:
- 学习导航
- 页面地图
- 浏览结构

### 12.5 Mobile rule
Do not hide orientation completely on mobile.
Compress it, but do not delete it.

---

## 13. Navigation System Acceptance Criteria

The navigation system should be considered successful if:
- a user can always identify current module and page
- a user can always see what else exists nearby
- a user can move forward by recommendation or jump sideways by explorer
- homepage remains guided rather than cluttered
- detail pages no longer feel isolated
- mobile still preserves orientation access

## 15. Short Handoff Summary

If another process reads only this file, the main instruction is:

> v0.2 must add a persistent spatial navigation layer.

The product should no longer rely only on cards, breadcrumbs, and next-step blocks.

It also needs:
- a left explorer on desktop
- a drawer-style explorer on mobile
- active current-location feedback
- visible nearby structure

That is the missing piece for making users feel they are inside a real learning product rather than a sequence of attractive pages.
