## 2. High-Level Build Goal

The goal is **not** to incrementally patch the current v0.1 static pages.

The goal is to:

> rebuild the experience architecture of openclaw-summary as a guided interactive learning product using a lightweight frontend framework while preserving static deployment and public accessibility.

The v0.2 rebuild should solve the following first-order problems:
- homepage routing weakness
- module pages being too similar
- lack of orientation and next-step guidance
- excessive hard page jumps
- lack of expandable previews
- repeated/templated above-the-fold structure

---

## 3. Confirmed Technical Direction

### Framework
- Recommended: **Astro**

### Why
- content-heavy site fit
- component model is sufficient without forcing app-heavy complexity
- easy static deployment
- compatible with interactive islands where needed
- good long-term maintainability for cards / accordions / navigation / metadata rendering

### Deployment target
- static deployment
- GitHub Pages compatibility should be preserved if practical
- public access should remain simple and frictionless

### Non-goals for this cycle
- no account system
- no backend CMS
- no dynamic database
- no server-dependent product architecture unless strictly necessary

---

## 4. Build Strategy

### Do not
- port old HTML page-for-page without structural redesign
- preserve repetitive KPI sidebars
- preserve oversized repeated module hero pattern
- treat the rebuild as primarily content expansion

### Do
- build a reusable page system
- implement page-type differentiation
- move content into data-driven structures where practical
- render metadata consistently
- support expansion/preview before full navigation

---

## 5. Required Page Types

The implementation should treat these as three distinct canonical page types.

### Type A — Homepage
Role:
- routing / orientation / first decision support

### Type B — Module Page
Role:
- expandable knowledge hub

### Type C — Detail Page
Role:
- self-contained reading page with strong wayfinding

If these 3 types are not clearly differentiated in code and layout, the rebuild will regress toward the v0.1 template problem.

---

## 6. Required Shared Systems

The following systems/components should exist early in the build.

### S1. Shared layout shell
Must support:
- global top navigation
- active module state
- responsive layout container
- footer

### S2. Breadcrumb system
Must support:
- Home > Module > Page
- configurable per page

### S3. Metadata rendering system
Should support:
- audience
- difficulty
- reading time
- tags
- prerequisites
- related pages
- next step

Not every field must be shown everywhere, but the schema should be designed up front.

### S4. Expandable card / accordion system
Must support:
- collapsed preview state
- expanded detail state
- CTA to full page

### S5. Next-step / related-page block
Must support:
- primary next action
- alternate related actions

### S6. Previous / next navigation block
Must support ordered reading sequences within modules.

## 7. Suggested Content/Data Architecture

The implementation should prefer structured content/data instead of hardcoding repeated page fragments.

### Recommended direction
Use a content-driven model such as:
- one content file per detail page
- one data object per module index page
- shared metadata fields across all content items

### Minimum useful schema per page
Suggested fields:
- `title`
- `slug`
- `module`
- `summary`
- `description`
- `audience`
- `difficulty`
- `readingTime`
- `tags`
- `prerequisites`
- `relatedPages`
- `nextStep`
- `prevPage`
- `nextPageOrdered`

### Why this matters
- avoids repetition
- improves consistency
- makes cards and page headers easier to render systematically
- enables later search/filter enhancements

## 10. Required Homepage Build Scope

The homepage implementation must include at minimum:

1. top navigation
2. simplified hero with primary and secondary CTA
3. three entry modes section
4. guided learning path section
5. fast-path shortcuts
6. module exploration section
7. shareable deep links section
8. FAQ/troubleshooting shortcut section

### Explicit anti-regression rule
Do not rebuild the homepage as another sequence of generic cards without route logic.

---

## 11. Required Module Page Build Scope

At minimum, each top-level module page should include:

1. breadcrumb + module header
2. “what this module helps with” block
3. recommended starting route
4. expandable topic cards
5. quick-answer / fast-path block
6. bridge to related modules

### Module-specific focus requirements

#### Getting Started
- stronger onboarding framing
- beginner route sequence
- confidence-building shortcuts

#### Concepts
- relation-driven grouping
- expandable concept previews
- concept relationship interaction or simplified relation panel

#### Setup
- task-flow feeling
- setup steps with success signals
- troubleshooting bridge

#### FAQ
- accordion / triage interaction
- symptom-driven entry

#### Use Cases
- scenario comparison logic
- capability metadata

#### Advanced
- grouped advanced topics by theme

---

## 12. Required Detail Page Build Scope

At minimum, each detail page should include:

1. breadcrumb + detail header
2. summary / quick answer block
3. main content area
4. common misunderstandings / pitfalls section when relevant
5. related pages block
6. next-step block
7. previous / next navigation

### Anti-regression rule
Do not use the old module-level hero template for detail pages.

Detail pages should feel like reading pages, not mini landing pages.

---

## 13. Visual System Implementation Guidance

### Keep
- dark, clean, slightly futuristic baseline
- readable card-based system
- strong spacing discipline

### Change
- reduce repetitive hero structures
- reduce generic badges used as decoration
- remove repeated KPI sidebars unless contextualized
- introduce module-level visual accents
- create clearer page-type-specific hierarchy

### Module identity direction
Each module should gain some combination of:
- accent color logic
- iconography
- header structure difference
- block rhythm difference
- interaction emphasis difference

This can remain subtle, but it must be noticeable.

---

## 14. Interaction Guidance

### Required first-cycle interactions
- expand / collapse topic cards
- expand / collapse FAQ groups
- route preview on key cards
- clear CTA hierarchy

### Acceptable implementation level
- simple, reliable interaction is preferred over flashy complexity
- no need for over-engineered animation in the first cycle

### Important
Interactivity must improve comprehension, not just decorate the interface.

---

## 15. Responsive / Mobile Guidance

The implementation should actively optimize for mobile, not merely collapse columns.

### Minimum responsive requirements
- smaller, tighter headers on mobile
- no oversized repeated hero patterns
- comfortable tap areas on expandable cards
- metadata remains readable when wrapped
- top nav remains usable without noisy line wrapping

### Preferred direction
- vertical path layouts on mobile
- reduced visual clutter above the fold

## 17. Acceptance Criteria for First Serious v0.2 Build

A first serious v0.2 build should be considered successful only if all of the following are true:

### Product experience
- homepage clearly routes users by stage, goal, or module
- module pages no longer feel like generic category lists
- detail pages feel self-contained and well-oriented
- major repeated hero/KPI problems from v0.1 are gone

### Navigation / flow
- breadcrumb exists on module and detail pages
- next-step guidance exists on detail pages
- previous / next navigation exists where reading order matters

### Interaction
- module pages support expandable topic previews
- FAQ supports accordion or equivalent triage behavior
- setup flow feels sequential and checkpoint-based

### Visual quality
- page types are visibly differentiated
- module identity is stronger than in v0.1
- typography hierarchy is calmer and more role-appropriate

### Technical quality
- codebase is componentized
- repeated content is not hardcoded in multiple places
- deployment remains static and accessible

---

## 18. Explicit Non-Goals for the First Build

The development process should not spend first-cycle time on:
- site search
- advanced filtering system
- user progress tracking
- CMS/editor tooling
- animation-heavy concept graphs
- perfect icon/illustration system
- polishing every single detail page before the system is stable

---

## 19. Recommended Definition of Done for the Development Process

The process can claim a good first handoff when:
- the new architecture is in place
- the three core page types are implemented well
- the four most important module experiences are rebuilt:
  - Getting Started
  - Concepts
  - Setup
  - FAQ
- the design problems classified as first-cycle P0 are materially improved
- the site is deployable and reviewable

---

## 20. Short Handoff Summary

If the next development process reads only this section, the main instruction is:

> Rebuild the product system, not just the pages.

Focus on:
- page-type differentiation
- orientation and next-step logic
- expandable previews
- data-driven reusable components
- static deployment with better UX

Avoid:
- copying the old template into Astro
- solving the wrong problem with more content
- preserving repetitive visual scaffolding from v0.1
