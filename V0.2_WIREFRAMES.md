# V0.2_WIREFRAMES.md — openclaw-summary

Last updated: 2026-03-31
Status: wireframe-level page blueprint for v0.2

---

## 1. Purpose

This document defines wireframe-level page structures for the v0.2 redesign.

It is meant to answer:
- what the main page types should look like
- what blocks each page must contain
- what order those blocks should appear in
- what interaction each block should support

This is not visual design in the pixel-perfect sense.

It is a **structural blueprint** for the next implementation process.

---

## 2. Wireframe conventions

Each wireframe below uses the same interpretation:

- **Block name** = structural section
- **Purpose** = why the block exists
- **Must include** = required content or components
- **Interaction** = expected behavior
- **Notes** = implementation / product guidance

---

## 3. Global page frame

All v0.2 pages should share a common outer frame, but not the same inner rhythm.

### Shared global frame
1. top navigation bar
2. optional breadcrumb / local nav layer
3. page-specific main content
4. contextual footer

### Shared top navigation should include
- brand / home link
- major modules
- active module state
- mobile collapse behavior

### Shared footer should evolve into
- related navigation
- project/repo link area
- version/update signal
- optional “continue learning” links

# 4. Homepage Wireframe

## Page type
Routing / orientation page

## Product job
Help a new or returning user quickly answer:
- what is this site
- how should I use it
- what is the best entry for me

---

## HOME-01 — Top Navigation

### Purpose
Create immediate global orientation.

### Must include
- logo / site title
- top-level modules
- current active state = Home
- mobile-friendly compact nav pattern

### Interaction
- standard navigation
- no heavy dropdown needed in first version

---

## HOME-02 — Hero / Primary Routing Block

### Purpose
Define the site in one sentence and push the user toward one primary action.

### Must include
- short site label
- one-sentence product definition
- one short support paragraph
- one primary CTA
- one secondary CTA

### Recommended content logic
- product definition: this is a Chinese learning product for OpenClaw
- primary CTA: 我是新手，带我开始
- secondary CTA: 按主题浏览

### Interaction
- CTA buttons go to beginner route or module browse route

### Notes
- keep this block much cleaner than v0.1
- do not overload with repeated generic badges
- any metadata shown here should actually help decisions

---

## HOME-03 — Three Entry Modes

### Purpose
Reduce choice anxiety by bundling entry decisions into 3 understandable modes.

### Must include
Three route groups:
1. 按阶段学习
2. 按目标进入
3. 按模块查阅

### Suggested structure
Each route group = one card or lane containing:
- title
- one-sentence explanation
- 3–6 child entry actions

### Interaction
- child items can be buttons, mini-cards, or expandable rows
- hover/tap should preview what the route gives the user

### Notes
- this is one of the most important differentiators from v0.1
- users should feel “I know how to start” after this block

---

## HOME-04 — Guided Learning Path

### Purpose
Turn the abstract learning path into a visible route.

### Must include
- sequence of 4–5 steps
- each step label
- what user gets from that step
- recommended pages
- estimated time / effort

### Interaction
- each step should expand
- expanded state reveals:
  - what this step teaches
  - key pages
  - expected outcome
  - continue button

### Notes
- can be vertical timeline or horizontal path depending on implementation quality
- vertical is likely easier for responsive design

---

## HOME-05 — “If You Only Want…” Fast Paths

### Purpose
Give users quick routes without reading the full homepage.

### Must include
Examples:
- 我只想 5 分钟了解 OpenClaw
- 我只想尽快跑起来
- 我只想看实际用途
- 我只想排查问题

### Interaction
- each option leads to a pre-curated page or path

### Notes
- helps impatient users and shared-link recipients

---

## HOME-06 — Module Exploration Layer

### Purpose
Allow open-ended exploration after routing is established.

### Must include
For each module:
- module title
- short explanation
- suitable audience
- “进入模块” action

### Interaction
- module cards may expand slightly to show “best first page” and “what’s inside”

### Notes
- this should come after routing and pathing, not before

---

## HOME-07 — Shareable Deep Links

### Purpose
Surface the most recommendable, independently useful pages.

### Must include
A small curated set such as:
- 什么是 OpenClaw
- Architecture
- First Setup Route
- Setup Troubleshooting

### Interaction
- simple card links
- can include short “when to share this” notes

---

## HOME-08 — Concept Relationship Preview

### Purpose
Tease the system nature of OpenClaw without overwhelming the user.

### Must include
- 5–7 core concept nodes
- one-sentence relation hints
- entry to full Concepts module

### Interaction
- clicking a node reveals connected concepts and why they matter

### Notes
- this can be modest in v0.2, does not need a complex graph engine at first

---

## HOME-09 — Troubleshooting / FAQ Shortcut

### Purpose
Catch users who arrived because they are already blocked.

### Must include
- quick question block
- links to most common problem areas

### Interaction
- can be simple segmented options such as:
  - 安装卡住了
  - 连接不上
  - 看不懂日志
  - 不知道某个现象是否正常

# 5. Module Page Wireframe Pattern

## Page type
Expandable knowledge hub

## Product job
Help a user understand:
- what this module is for
- what inside this module matters most
- what they should read first
- whether they should stay in preview mode or open full pages

This pattern applies to:
- Getting Started
- Concepts
- Setup
- Use Cases
- Advanced
- FAQ

Each module will vary in flavor, but should share a core structure.

---

## MODULE-01 — Breadcrumb + Module Header

### Purpose
Instantly orient the user.

### Must include
- breadcrumb
- module label
- module title
- one short purpose line
- optional audience tags

### Interaction
- breadcrumb links back upward

### Notes
- this should replace the oversized landing-page-style hero used in v0.1 for most module pages

---

## MODULE-02 — “What This Module Helps You Do” Block

### Purpose
Explain the module’s job in plain language.

### Must include
- 2–4 bullet outcomes
- who this module is for
- when to use it

### Notes
Examples:
- Getting Started = help first-time users begin without confusion
- Concepts = help users understand how the system fits together
- Setup = help users complete first successful setup
- FAQ = help users diagnose common known issues quickly

---

## MODULE-03 — Recommended Starting Route

### Purpose
Prevent the module from becoming a flat list.

### Must include
- “Start here” recommendation
- “Then read…” sequence
- “If you only read one thing…” shortcut

### Interaction
- route items can be step cards or a mini-timeline

---

## MODULE-04 — Expandable Topic Cards

### Purpose
Provide preview before full navigation.

### Must include for each topic
- title
- one-sentence summary
- audience
- difficulty
- estimated reading time
- prerequisite pages (if any)
- recommended next pages
- open full page action

### Interaction
Collapsed state:
- title + summary + key metadata

Expanded state:
- why this matters
- common misunderstanding or use case
- links to related topics
- CTA to full page

### Notes
- this is the core module-page behavior of v0.2

---

## MODULE-05 — “Only Need a Quick Answer?” Block

### Purpose
Support users who do not want to inspect the full module.

### Must include
- one or two rapid entry points
- a short explanation of when to use them

### Examples
- Concepts: 先看 Architecture
- Setup: 先走最短配置路线
- FAQ: 先按症状查入口

---

## MODULE-06 — Related Module Bridge

### Purpose
Prevent silo behavior between modules.

### Must include
- 2–3 cross-module recommendations
- explanation of why user might go there next

### Example
- from Concepts → Setup
- from Setup → FAQ
- from Getting Started → Concepts

---

# 6. Module-Specific Variations

---

## 6.1 Getting Started Module Wireframe

### Identity
Onboarding hub

### Block emphasis
1. breadcrumb + lightweight header
2. “this module is for first-time users” block
3. recommended beginner route
4. quick confidence-building checklist
5. expandable beginner topic cards
6. fast path: “only have 10 minutes?”
7. next move: go to Concepts or Setup

### Special interaction ideas
- estimated time for each beginner step
- “minimum first success path” box
- confidence checklist such as:
  - 我知道它是什么
  - 我知道先看哪几个词
  - 我知道第一轮不需要全懂

---

## 6.2 Concepts Module Wireframe

### Identity
Relationship map / knowledge system hub

### Block emphasis
1. breadcrumb + lightweight header
2. why relationships matter
3. recommended concept reading order
4. concept relationship view
5. expandable concept cards
6. grouped related clusters
7. bridge to Setup / Advanced

### Special interaction ideas
- concept nodes that reveal linked concepts
- grouped clusters such as:
  - Core system
  - Execution layer
  - Continuity layer
  - Workflow layer

---

## 6.3 Setup Module Wireframe

### Identity
Guided task flow

### Block emphasis
1. breadcrumb + lightweight header
2. “what success looks like” block
3. shortest recommended path
4. setup progress flow
5. expandable step cards
6. common failure checkpoints
7. bridge to troubleshooting

### Special interaction ideas
- step state visualization
- per-step success signals
- “stuck here?” links in each expanded step

---

## 6.4 FAQ Module Wireframe

### Identity
Problem triage surface

### Block emphasis
1. breadcrumb + lightweight header
2. symptom-based quick chooser
3. issue category accordion
4. top known issue clusters
5. links to full FAQ pages
6. “if this is not your issue, go here next” block

### Special interaction ideas
- accordion-first UI
- issue cards that show symptoms + likely cause + confidence note
- categories such as:
  - setup
  - connectivity
  - files / memory
  - ws/http fallback
  - behavior / safety

---

## 6.5 Use Cases Module Wireframe

### Identity
Scenario showcase

### Block emphasis
1. breadcrumb + lightweight header
2. “what OpenClaw can actually do” block
3. browse by user goal
4. expandable scenario cards
5. compare scenarios by capability set
6. bridge to setup / concepts

### Special interaction ideas
- scenario comparison chips
- “best for” labels
- “what capabilities this uses” metadata

---

## 6.6 Advanced Module Wireframe

### Identity
Depth map / advanced topic board

### Block emphasis
1. breadcrumb + lightweight header
2. “who this is for” block
3. grouped advanced topic categories
4. expandable advanced topic cards
5. “recommended first advanced topics” shortcut
6. bridge back to concepts and best practices

### Special interaction ideas
- group by theme:
  - session & memory
  - workflow design
  - safety & operations
  - debugging & logs

# 7. Detail Page Wireframe Pattern

## Page type
Self-contained learning page

## Product job
Explain one topic deeply enough while preserving orientation and forward motion.

---

## DETAIL-01 — Breadcrumb + Local Context Header

### Purpose
Instantly locate the user in the site.

### Must include
- breadcrumb
- module label
- page title
- optional subtitle
- difficulty + reading time + audience tags

### Notes
- much lighter than a landing-page hero
- should feel like a reading interface, not a campaign page

---

## DETAIL-02 — Quick Answer / Page Summary Box

### Purpose
Allow fast understanding before deep reading.

### Must include
- one short answer
- why it matters
- who should read this
- prerequisite / next step

### Interaction
- can be always visible or collapsible

### Notes
- this block is especially important for shared-entry visitors

---

## DETAIL-03 — Main Explanation

### Purpose
Deliver the core content.

### Must include
- structured sections
- clear subheads
- examples where useful
- relationship explanation where relevant

### Notes
- keep short answer first, deeper explanation second

---

## DETAIL-04 — Common Misunderstandings / Pitfalls

### Purpose
Reduce wrong mental models.

### Must include
- 2–5 common mistakes or false assumptions

### Notes
- especially valuable for concept, setup, and troubleshooting pages

---

## DETAIL-05 — Related Pages

### Purpose
Help the user broaden understanding intelligently.

### Must include
- 2–4 related pages
- one-sentence reason for each relationship

---

## DETAIL-06 — Recommended Next Step

### Purpose
Preserve flow and reduce drop-off.

### Must include
- one primary next page
- one or two alternate next pages

### Interaction
- clear primary action button
- optional secondary route links

---

## DETAIL-07 — Previous / Next Navigation

### Purpose
Support continuous reading.

### Must include
- previous page
- next page

### Notes
- sequence can be module-defined rather than purely alphabetical

---

# 9. Component-Level Wireframe Notes

## 9.1 Expandable Topic Card

### Collapsed state
- title
- one-sentence summary
- metadata row

### Expanded state
- why it matters
- common misunderstanding / use case
- prerequisites
- related pages
- primary CTA = open full page

---

## 9.2 Metadata Row

### Preferred fields
- difficulty
- reading time
- audience
- tags

### Notes
- do not overload every card with every field
- use only what improves decisions

---

## 9.3 Recommended Next Step Block

### Structure
- title: 下一步建议
- primary recommendation
- optional alternate routes
- short explanation for each

---

## 9.4 Quick Path Block

### Structure
- title such as “如果你只想……”
- 2–4 short options
- each option maps to a curated route or page

---

# 10. Mobile Behavior Guidance

## Rules
- compress headers significantly on mobile
- avoid large duplicated hero structures
- prefer vertical path layout
- expandable cards should remain comfortable to tap
- metadata should wrap cleanly and remain scannable
- breadcrumb can be shortened if necessary but should not disappear entirely

---

- top nav

- footer

- next-step block

- expandable card
- accordion
- route block
- timeline / step list

## Step 3
Implement three canonical page types:
- homepage
- module page
- detail page

## Step 4
Apply module-specific variations:
- getting-started
- concepts
- setup
- faq

## Step 5
Apply remaining module styles:
- use-cases
- advanced

---

## 12. Short Handoff Summary

If another process reads only this file, the most important takeaway is:

> v0.2 should stop treating every page like a variation of the same marketing-style template.

Instead:
- homepage = routing page
- module page = expandable knowledge hub
- detail page = self-contained reading page with strong orientation

This page-type separation is essential to making the product feel complete.
