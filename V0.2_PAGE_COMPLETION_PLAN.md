---

## 1. Why this document exists

The project should not stop at improving only a few key pages.

If only homepage + a handful of important modules receive the new system, the product will still feel incomplete.

Therefore v0.2 needs a page-completion plan that answers:
- which pages must be fully rebuilt first
- which pages can use lighter patterns temporarily
- how all sitemap pages should fit into the new system
- how to avoid a split between “good pages” and “leftover pages”

---

## 2. Completion strategy principle

The product should be rebuilt as a system, but not every page needs the same depth in the first serious pass.

Use three completion levels.

### L-A — Full Pattern Page
A page that fully uses the v0.2 detail-page model.

Must include:
- breadcrumb
- explorer context
- full detail header
- summary box
- structured body
- misunderstandings / pitfalls where relevant
- related pages
- next step
- prev / next

### L-B — Standard Structured Page
A page that uses the core reading pattern, but with lighter content depth.

Must include:
- breadcrumb
- detail header
- summary
- structured sections
- next step
- related pages where possible

### L-C — Transitional Structured Page
A page that is already inside the new system, but still waits for deeper content expansion.

Must include:
- breadcrumb
- page header
- short summary
- brief structured body
- next step

The key rule is:

> Even transitional pages must feel like part of the same product system.

## 3. Completion targets by page type

### Homepage
Target level:

### Module pages

- full module-hub pattern for all top-level modules

### Detail pages
Target level:
- mixed by priority

## 7. Module Completion Requirements

Every top-level module must be considered incomplete until all of the following are true.

### Getting Started completion rule
Complete when:
- module hub is strong
- all 3 child pages are in the new system
- newcomer can go from orientation to setup without friction

### Concepts completion rule
Complete when:
- relation-driven module hub exists
- core pages (architecture / session / gateway / skills / tools / memory) are complete
- secondary concept pages are discoverable through explorer and module grouping

### Setup completion rule
Complete when:
- setup hub feels like a task flow
- all major steps are represented
- troubleshooting bridge is strong
- success-signal logic is present in relevant detail pages

### FAQ completion rule
Complete when:
- FAQ hub behaves like triage surface
- setup / connectivity / files / fallback are covered
- question routing is stronger than raw list browsing

### Use Cases completion rule
Complete when:
- module hub supports scenario exploration
- major scenario categories are present
- each scenario page is connected back to concepts/setup

### Advanced completion rule
Complete when:
- module hub supports grouped advanced exploration
- strongest advanced topics are first-class
- deeper/long-tail topics are still navigable even if lighter initially

## 8. Completion by User Journey

The product should also be judged by journey completeness, not only by page count.

### Journey 1 — “I am new”
Must be complete across:
- homepage
- getting started
- what is openclaw
- how to learn
- architecture
- install
- configuration
- start gateway
- first message
- setup FAQ

### Journey 2 — “I want to understand the system”
Must be complete across:
- homepage
- concepts hub
- architecture
- session
- gateway
- skills
- tools
- memory

### Journey 3 — “I am blocked and need help”
Must be complete across:
- homepage FAQ shortcut
- FAQ hub
- setup FAQ
- connectivity FAQ
- files & memory FAQ
- ws/http fallback FAQ

### Journey 4 — “I want to see value and real scenarios”
Must be reasonably complete across:
- homepage scenario entry
- use-cases hub
- personal assistant
- messaging platforms
- google workspace / notes-reminders / github workflow as available

This journey-based check helps prevent local completion without product-level completeness.

## 9. Content-Depth Strategy for Remaining Pages

### Rule 1
Do not leave pages as raw placeholders outside the new system.

### Rule 2
If a page is not fully written yet, it should still:
- have a proper header
- have a concise summary
- explain why it matters
- suggest where to go next

### Rule 3
Depth can vary, but product coherence should not.

### Wave 1 — Backbone completion
Goal:

### Wave 2 — Scenario and advanced breadth
Goal:
- bring Use Cases and Advanced into the same system quality

### Wave 3 — Long-tail completeness and polish
Goal:
- deepen secondary pages

- refine mobile and explorer behavior

## 11. Completion Checklist per Detail Page

Before a detail page is considered “inside v0.2,” it should pass this checklist.

### Minimum checklist
- inside the new layout shell
- breadcrumb present
- explorer context present where applicable
- title and summary are stable
- page has a clear purpose
- next step exists
- page is linked into related module navigation

### Strong-page checklist
- quick answer present
- misunderstandings/pitfalls present where relevant
- related pages present
- prev/next present where useful
- content sections are structured and readable

---

## 12. Risk to avoid

### Risk 1
Only the main pages feel polished, while child pages still feel leftover.

### Risk 2
All pages get rebuilt structurally, but content depth is too inconsistent without explanation.

### Risk 3
Use Cases and Advanced remain obviously less mature than the core backbone.

### Risk 4
Explorer and navigation only work for some modules, causing uneven product trust.

Mitigation:
- define completion levels explicitly
- track pages by level
- upgrade modules systematically rather than randomly

## 14. Short Handoff Summary

If another process reads only this file, the key message is:

> v0.2 should not stop at fixing a few flagship pages.

The product must avoid a split between:
- high-quality core pages
and
- abandoned long-tail pages.

Use a tiered completion strategy:
- backbone pages = full pattern
- important expansion pages = standard structured
- long-tail pages = transitional structured

That way the entire sitemap can enter the new product system without pretending every page needs identical depth immediately.
