# V0.2_REDESIGN_BRIEF.md — openclaw-summary

Last updated: 2026-03-31
Status: confirmed redesign brief for next version

---

## 1. Why v0.2 exists

The current v0.1 successfully proved that the product direction is valid:
- Chinese-only positioning is clear
- the topic coverage is broad enough
- the information architecture is fundamentally correct
- the site is already shareable and publicly accessible

However, real usage feedback exposed an important gap:

> v0.1 is structurally closer to a well-organized static documentation site than to a guided learning product.

The main issue is **not lack of content**, but lack of:
- orientation
- wayfinding
- progressive disclosure
- low-friction interaction
- confidence-building learning flow

So v0.2 should not be framed as “add more pages.”

It should be framed as:

> Transform the project from a multi-page content site into a guided interactive learning product.

---

## 2. Confirmed redesign decision

### Product-level decision
- Keep the project as a public, shareable Chinese learning site
- Keep independent page URLs for important topics
- Keep the 7-module information architecture as the core knowledge model

### Technical decision
- Migrate from pure static hand-authored multi-page HTML to a **lightweight frontend framework**
- Prefer a framework route that still supports **static deployment**
- Public access must remain as easy as v0.1

### Recommended implementation direction
Preferred order:
1. Astro
2. Next.js with static export
3. Vite + React

Current recommendation:

> Use **Astro** as the default v0.2 direction.

Why Astro:
- very suitable for content-heavy learning sites
- supports componentized UI without forcing a heavy app model
- easy to keep static deployment and public accessibility
- good fit for collapsible cards, content islands, navigation components, and shareable routes
- simpler than a full app-heavy framework for this project stage

---

## 3. Core diagnosis from v0.1

### D1. Homepage feels like a directory, not a guided entry
Users can see many sections, but they are not strongly guided toward the best starting action.

### D2. Cards behave like link boxes, not interaction units
Most cards only lead to another page. This creates too much jump friction and too little preview.

### D3. There is not enough orientation inside the product
Users do not consistently know:
- where they are
- what this page is for
- what they should read next
- how this page fits into the larger learning journey

### D4. The learning path exists in writing, but not strongly enough in UI
The product principles already emphasize pathing and progression, but the interface does not yet embody them strongly enough.

### D5. Multi-page architecture is still valuable, but needs a continuous browsing layer
Independent URLs remain useful for sharing and deep reading, but users also need smoother in-page exploration.

---

## 4. v0.2 design goals

v0.2 should make the site feel:
- more guided
- more interactive
- more confidence-building
- less jumpy
- more like a learning product than a docs mirror

Success means:
- a new user can immediately choose a suitable learning path
- users understand what each module/page is for before entering it
- cards can reveal useful information before forcing a navigation decision
- pages clearly tell users what to do next
- the site becomes easier to browse continuously without losing shareability

---

## 5. v0.2 experience principles

### P1. Guide before you expose
Do not present too many equal-weight choices too early.

### P2. Reduce navigation anxiety
A click should feel safe. Users should know what they are entering before they leave the current context.

### P3. Keep independent pages, but add progressive disclosure
Use expansion, preview, and local interaction before sending users into full-page reading.

### P4. Every page must answer 3 questions
- Where am I?
- Why am I here?
- What should I read next?

### P5. Learning flow beats visual density
Even a beautiful interface fails if the user cannot tell what to do next.

## 6. Homepage redesign

The homepage should become a **routing and confidence-building page**, not a full content showcase.

### 6.1 New homepage role
The homepage should:
- explain what the site is
- explain how to use the site
- route users by stage, goal, or module
- highlight one recommended action first

### 6.2 New homepage structure

#### Section A — Hero
Must include:
- one-sentence product definition
- one primary CTA
- one secondary CTA
- short explanation of how the site should be used

Recommended CTA structure:
- Primary CTA: 我是新手，带我开始
- Secondary CTA: 按主题浏览

#### Section B — Three entry modes
Replace the current many-parallel-entry feeling with 3 clearer modes:

1. 按阶段学习
   - 新手入门
   - 建立心智模型
   - 完成配置
   - 进入进阶
   - 排错

2. 按目标进入
   - 我想先搞懂 OpenClaw
   - 我想尽快部署成功
   - 我想看实际场景
   - 我想排查常见问题

3. 按模块查阅
   - Getting Started
   - Concepts
   - Setup
   - Use Cases
   - Advanced
   - FAQ

#### Section C — Guided learning path
The learning path should become visually stronger.

Recommended interaction:
- path timeline or route map
- each step expandable
- each step shows:
  - what you will learn
  - recommended pages
  - estimated reading time
  - expected outcome

#### Section D — Free exploration layer
Keep module cards and concept preview, but move them after the guided entry sections.
This makes exploration optional instead of mandatory.

#### Section E — Shareable deep links
Show a small block of highly shareable pages, such as:
- what is OpenClaw
- architecture
- first setup route
- common troubleshooting

## 7. Module page redesign

All top-level module pages should stop behaving like simple jump lists.

They should become:

> expandable knowledge hubs

### 7.1 Required interaction pattern
Each major topic card on a module page should support expansion.

Expanded state should show:
- one-sentence summary
- why this matters
- who should read this
- difficulty
- estimated reading time
- prerequisite pages
- recommended next pages
- a button to open the full page

### 7.2 Priority modules for this pattern
Apply first to:
- `/getting-started`
- `/concepts`
- `/setup`
- `/faq`

Then extend to:
- `/use-cases`
- `/advanced`

### 7.3 Module page content blocks
Every module page should have:
- module introduction
- recommended reading order
- expandable topic cards
- “if you only want one thing from this module” shortcut
- next-step navigation

---

## 8. Detail page redesign

Independent pages remain important, but they must become more self-orienting.

### 8.1 Required page-level wayfinding components
Every detail page should include:
- breadcrumb
- current module indicator
- page purpose statement
- recommended next step
- related pages
- previous / next navigation

### 8.2 Optional but strongly recommended components
- reading time
- difficulty label
- “good for beginners / intermediate / advanced” badge
- “you are here in the learning path” marker
- collapsible “quick answer” and “deeper explanation” sections

### 8.3 Detail page reading model
Preferred structure:
1. short answer
2. why it matters
3. deeper explanation
4. common misunderstandings
5. related concepts/pages
6. what to read next

---

## 9. Global navigation redesign

### 9.1 Navigation problems to solve
The product needs stronger orientation and less disorientation after entering deep pages.

### 9.2 Required global navigation upgrades
- persistent active-state top navigation
- breadcrumb on all subpages
- previous / next page links
- recommended next page block
- module-level local navigation

### 9.3 Optional navigation upgrades
- mobile-friendly module drawer
- sticky in-page table of contents for long pages
- floating “back to learning path” action

---

## 10. Card system redesign

The card system is now a core product component, not just a layout style.

### 10.1 Card roles
Cards should support different roles:
- route card
- topic preview card
- expandable summary card
- scenario card
- troubleshooting card

### 10.2 Standard card metadata
Cards should support these optional metadata fields:
- audience
- difficulty
- estimated reading time
- prerequisites
- tags
- recommended sequence number

### 10.3 Behavioral rule
If a card is meant to introduce a topic, it should preferably reveal a summary first before forcing a full-page navigation.

---

## 11. Concept map redesign

The concept map should evolve from a static preview area into a semi-interactive or interactive structure.

### v0.2 minimum goal
A user can click a core concept and reveal:
- related concepts
- one-sentence relationship notes
- where to learn more

### Future ideal state
A fully interactive relationship map with highlighted nodes and guided exploration.

For v0.2, a simplified interactive relationship panel is sufficient.

---

## 12. Information architecture updates

The top-level seven modules remain valid and should stay.

However, the presentation logic should shift from:
- broad parallel exposure

to:
- guided entry first
- expandable preview second
- full page reading third

This is a presentation-layer redesign, not a full knowledge-model reset.

---

## 13. Content strategy updates

### 13.1 Do not solve the current problem by only writing more
The main bottleneck is product guidance and interaction, not page count.

### 13.2 Content should support progressive disclosure
Each content unit should ideally exist at 3 levels:
- preview
- page summary
- full explanation

### 13.3 Metadata becomes part of content design
Difficulty, reading time, audience, prerequisites, and next steps should be treated as first-class content fields.

---

## 14. Technical implications for v0.2

The redesign introduces interaction and reusable UI requirements that are no longer ideal for the current pure static hand-authored approach.

### Required capabilities
- reusable layout and card components
- collapsible sections and expandable cards
- route-aware active navigation
- module-level data-driven rendering
- metadata-driven cards and page summaries
- easy future support for search/filter/path state

### Therefore
The next implementation should use a lightweight component-based frontend architecture with static deployment output.
